HTTP 메시지

HTTP 프로토콜에 따라 주고 받는 데이터의 형식
구분 요청 응답 형식
시작줄 <메서드> <요청Url> <버전> <버전> <상태코드> <사유 구절> 스페이스로 구분되어있다.
헤더 <헤더> <헤더> 헤더는 이름:값 쌍으로 구성된다
본문 <엔티티 본문> <엔티티 본문> 마지막줄을 crlf로 설정하여 종료됬음을 표시
<메서드> : 서버에 요청하는 내용이 무엇인지를 정의한다
  • GET: 서버에 리소스를 요청한다.
  • POST: 서버에 처리해야할 데이터를 보낸다.
  • HEAD: 서버에서 메시지의 헤더만 받는다.(리소스를 체크할 수 있다)
  • PUT: 서버에 요청 메시지의 본문을 저장한다.(서버에 리소스를 생성한다.사용자 인증)
  • TRACE: 메시지의 경로과정을 추적한다.(PROXY,GEATEWAY...) 진단을 위해서 사용된다.
  • OPTIONS: 서버가 어떤메서드를 수행할 수 있는지 체크한다.
  • DELETE: 서버에서 문서를 제거한다.단 HTTP명세에는 서버에서 클라이언트에게 알리지 않고 요청을 무시하는 것을 허용한다.
  • 확장:LOCK (리소스 잠금) ,MKCOL(문서를 생성),COPY(서버에 있는 리소스 복사),MOVE(서버에 있는 리소스를 옮긴다.)
<상태코드> : 서버가 클라이언트에게 요청에 대한 결과를 코드로 알려준다.(사유 구절보다 코드 의미를 파악한다)
  • 100~199: 정보성코드
    • 100: 요청의 시작 부분 일부가 서버에서 받아들여젔고 클라이언트는 나머지를 계속 보내야함,클라이언트 개발자는 <엔티티 본문>이 없을경우 헤당코드를 보내면 안된다. 그리고 경우에 따라서 서버가 헤당코드를 리턴하지 않을 수 있으므로 고려해서 개발해야함
    • 101: 서버가 프로토콜을 변경하였음을 의미
  • 200~299: 성공 상태 코드
    • 200: 요청은 정상이고 엔터티 본문은 리소스를 포함하고 있음
    • 201: 요청에 대해서 서버는 개체를 생성했고 생성한 리소스에 대한 정보가 있는 Location 헤더 와 참조할 수 있는 여러 Url를 본문에 포함해야 한다.
    • 202: 요청은 받아 들여젔지만 서버가 동작을 수행하지 않았음
    • 203: 요청은 정상이지만 엔터티 헤더에 들어있는 정보가 리소스 사본에서 왔음을 표시 (200code 파생)
    • 204: 엔터티 본문은 포함하지 않는다.(브라우저 폼 리프래쉬)
    • 205: 브라우저에게 현재 페이지 HTML 폼에 값을 비우라고 요청
    • 206: 부분,범위에 대해서 요청이 성공했음을 의미
  • 300~399: 리다이렉션 상태 코드, 요청한 리소스의 본문 대신 위치를 알려준다. 브라우저가 자동적으로 재호출
    • 300: 클라이언트가 동시에 여러 리소스를 가리키는 URL을 요청한 경우,그 리소스의 목록과 함꼐 반환된다.
    • 301: 요청한 url이 옮겨젔을때 사용한다.
    • 302: 요청한 url이 임시적으로 옮겨젔을때 사용한다.(HTTP/1.0명세)
    • 303: POST요청에 대한 응답으로 리소스위치가 변경됬을때 사용
    • 304: 클라이언트가 조건부 헤더로 요청한 경우, 리소스변경되지 않아서 반환활 필요가 없을 경우
    • 305: 리소스가 프락시를 통해서 접근되야함, 프락시 위치를 알려준다.
    • 307:요청한 url이 임시적으로 옮겨젔을때 사용한다.(HTTP/1.1명세)
  • 400~499: 클라이언트 에러 상태 코드
    • 400: 잘못된 요청
    • 401: 리소스를 얻기전에 인증이 필요함
    • 403: 서버에 의해서 요청이 거부됨
    • 404: 요청한 URL을 찾을 수 없음
    • 405: 요청한 URL에 지원되지 않는 메서드
    • 406: 클라이언트가 요청한 엔티티 본문의 형식을 서버가 응답할수 없을 경우
    • 407: 인증요구를 프락시를 통해서 해야됨
    • 408: 요청에 시간이 많이 걸리는 경우
    • 409: 요청이 서버쪽 리소스에 충돌을 발생시킬 경우가 있는 경우 서버는 그 사유를 적어서 응답
    • 410: 요청한 URL이 과거 서버에 있었으나 현재는 없음
    • 411: 서버가 클라이언트에게 Content-length를 포함해서 다시 보내라고 요청
    • 412: 클라이언트가 보낸 조건부 요청중 하나가 실패
    • 413: 서버가 처리할수 있는 용량보다 큰 요청이 온경우
    • 414: Url의 길이가 너무 긴 경우
    • 415: 서버가 지원하지 않는 mime type인 경우
    • 416: 요청 메시지가 리소스의 특정범위를 요청했으나 범위가 잘못된 경우
    • 417: Expect 요청 헤더에 응답하지 못할경우
  • 500~599 :클라이언트는 적절한 요청을 보냈으나 서버측 오류
    • 500: 서버 내부적 에러
    • 501: 서버의 능력을 넘는 요청 (ex:서버가 지원하지 않는 메서드 요청)
    • 502: 프락시나 게이트웨어 처림 요청 응답 연쇄가 일어나야 할때 중간에 가짜 응답을 만난 경우
    • 503: 서버에 일시적인 장애 추후 응답가능함
    • 504: 다른 서버에게 요청을 보내고 응답을 기다리다가 타임아웃이 발생한 경우
    • 505: http 버전 오류
<헤더> : 일반 정보와 메서드에 사용되는 정보들로 구성되어 있다.
  • 일반 헤더: 일반 정보 제공
    • Connection:요청/응답에 대한 옵션을 정할수 있다.
    • Date:메시지 생성일시 제공.
    • MIME-Version:발송자가 사용한 mime의 버전을 제공.
    • Trailer chunked transfer:인코딩으로 인코딩된 메시지의 끝 부분에 위치한 헤더들의 목록을 나열한다.
    • Transfer-Encoding:수신자에게 안전한 전송을 위해 어떤 메시지에 어떤 인코딩이 사용됬는지 제공
    • Upgrade:발송자가 업그레이드 하길 원하는 새 버전이나 프로토콜을 알려준다.
    • Via:이 메시지가 어떤 중개자(프록시,게이트웨이)를 거쳐왔는지 보여준다
  • 일반 캐시 헤더: 캐시에 대한 정보 제공
    • Cache-Control:메시지와 함계 캐시 지시자를 전달.
    • Pragma:메시지와 함꼐 지시자를 전달하는 다른 방법(deprecated).
  • 요청 헤더: 요청 메시지에서만 의미가 있다.
    • Client-IP:요청자 ip를 서버에 제공.
    • From:요청자의 메일 주소를 제공
    • Host:요청 대상인 서버의 호스트명과 포트
    • Refer:현재 요청 uri가 들어있었던 문서의 url을 제공
    • User-Agent:요청을 보낸 애플리케이션 이름
    • UA-Color,UA-CPU,UA-Disp,UA-OS,UA-Pixels: 클라이언트 피시 정보
    • Accept,Accept-Charset,Accept-Encodign,Accept-Language: 클라이언트 원하는 미디어,문자집합,인코딩,언어를 서버에 제공
    • Authorization:클라이언트가 서버에게 제공하는 인증
    • Cookie:클라이언트가 서버에게 토큰을 전달할때 사용
    • Cookie2:클라이언트가 지원하는 쿠키의 버전을 알려줄때 사용
    • Max-Forwards: 다른 프락시나,게이트웨이로 전달되는 최대 횟수 제한
    • Proxy-Authorization: 프락시에서 사용되는 인증
    • Proxy-Connection: 프락시에서 사용되는 요청과응답 설정
  • 응답 헤더: 응답 메시지들이 갖는 헤더
    • Age:응답이 얼마나 오래됬는지
    • Public:서버가 특정 리소스에 지원하는 요청 메서드 목록
    • Retry-After:현재 리소스가 사용불가 일때 언제부터 가능한지에 대한 일시정보
    • Server:서버 애플리케이션의 이름과 버전
    • Warnig:사유 구절보다 더 자세한 메시지
  • 응답 보안 헤더:
    • WWW-Authenticate: 서버에서 클라이언트로 보낸 인증요구의 목록
    • Proxy-Authenticate: 프락시에서 클라이언트로 보낸 인증요구의 목록
    • Set-Cookie:서버가 클라이언트를 인증 할 수 있도록 클라이언트측 토큰을 설정하기 위해 사용
    • Set-Cookie2:서버가 클라이언트를 인증 할 수 있도록 클라이언트측 토큰을 설정하기 위해 사용
  • 엔터티 헤더:
    • Allow: 이 엔터티에 대해 수행될 수 있는 요청 메서드를 나열한다
    • Location: 클라이언트에게 엔티티가 실제로 어디에 있는지 알려준다
    • ETAG: 엔터티에 대한 엔터티 태그
    • Expires: 엔터티가 유효하지 않아서 원본을 다시 받아와야하는 경우
    • Last-Modified:가장 최근에 변경된 일시
    • Content-base:기저 url
    • Content-Encoding,Language,Length,Location,MD5,Range,Type:콘탠츠의 인코딩 ,언어 ,길이, 실제로 있는 위치, MD5체크섬, 전체 리소스에서 이 컨첸츠가 속하는 범위를 바이트단위로 표시 ,객체 타입
